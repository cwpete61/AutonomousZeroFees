# Recovery and Restore Runbook

This document outlines the procedures for recovering the Autonomous Web Agency platform from data loss, corruption, or catastrophic failure.

## 1. Database Restoration (Prisma/Postgres)

In the event of database corruption or accidental deletion, follow these steps to restore from a backup.

### Identifying Backups
Backups are automatically generated by the `backup-service` and stored in the `/var/backups/orbis` directory on the production host (mapped from the `backups` volume).

### Restoration Procedure
1. **Stop the Application**:
   ```bash
   docker-compose -f docker-compose.prod.yml stop api workers
   ```

2. **Restore the Database**:
   ```bash
   # Copy the latest backup SQL file into the postgres container
   docker cp /path/to/backup.sql postgres:/tmp/restore.sql

   # Run the restore command
   docker-compose -f docker-compose.prod.yml exec postgres psql -U agency_prod -d agency -f /tmp/restore.sql
   ```

3. **Verify and Restart**:
   ```bash
   docker-compose -f docker-compose.prod.yml start api workers
   ```

## 2. Code Rollback

If a new deployment introduces critical bugs (e.g., the `AuditInterceptor` issue previously encountered):

1. **Identify the Last Stable Version**:
   ```bash
   git log --oneline -n 10
   ```

2. **Rollback to Stable Commit**:
   ```bash
   git checkout <stable-hash>
   ```

3. **Re-deploy Everything**:
   ```bash
   docker-compose -f docker-compose.prod.yml up -d --build
   ```

## 3. Redis State Clear (Optional)

If the queue state (Bull) becomes corrupted or jammed:

1. **Flush Redis**:
   ```bash
   docker-compose -f docker-compose.prod.yml exec redis redis-cli flushall
   ```
   *Warning: This will clear all pending jobs and active tasks.*
